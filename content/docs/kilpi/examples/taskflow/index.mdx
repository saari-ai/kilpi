title: Taskflow Example
description: A worked example applying the CSF methodology to a real product
---

Taskflow is a fictional SaaS project management tool used as a worked example across all six NIST CSF 2.0 functions. The fictional organisation behind Taskflow also operates a billing portal, marketing site, and build pipeline — four components that exercise the full per-component assessment structure.

The purpose of this example is to show how the CSF methodology produces concrete, traceable security artefacts for a real organisation. Every threat maps to an asset. Every control maps to a threat. Every monitoring rule maps to a control. Every response playbook maps to a detection trigger. And every subcategory ID ties the whole chain back to the NIST CSF 2.0 framework.

The [Component Register](/docs/kilpi/examples/taskflow/component-register) lists every component in the fictional organisation. The org-level sections (Govern, Respond, Recover) are completed once. Each component then gets its own Identify, Protect, and Detect assessment.

## Org-level sections

| Section | Scope |
|---|---|
| [Govern](/docs/kilpi/examples/taskflow/govern) | Organizational context, risk strategy, roles, policy, oversight, and supply chain risk |
| [Component Register](/docs/kilpi/examples/taskflow/component-register) | All assessed components, owners, classification, and criticality |
| [Respond](/docs/kilpi/examples/taskflow/respond) | Incident management, analysis, reporting, and mitigation |
| [Recover](/docs/kilpi/examples/taskflow/recover) | Recovery plan execution and recovery communication |

## The traceability chain

The example demonstrates a full traceability chain across NIST CSF 2.0:

```
Asset (ID.AM) → Threat (ID.RA) → Control (PR.*) → Monitoring (DE.*) → Response (RS.*) → Recovery (RC.*)
```

Every table in the category pages includes references to the specific asset IDs (SA-TF-*, SA-BL-*, SA-MK-*, SA-CI-*), threat IDs (ST-TF-*, ST-BL-*, etc.), and control IDs (SC-TF-*, SC-BL-*, etc.) that connect the chain. This is the mechanism that connects product-level security work to the framework.
