## Our methodology

Platform security is where abstract controls become concrete configuration. This is the most technical PR category â€” it covers hardened baselines, patching, logging, and secure development practices.

---

### PR.PS-01: Configuration management

Define a hardened baseline for every platform component. Start from a security benchmark, not from defaults. Document deviations with justification.

| Component | Baseline source | Key settings | Deviation notes |
|---|---|---|---|
| *e.g. Web application* | *Security headers (CSP, HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy)* | *CSP: script-src self; HSTS: max-age 31536000 includeSubDomains* | *None* |
| *e.g. TLS* | *Mozilla TLS guidelines* | *TLS 1.2 min, ECDHE key exchange, no RC4/3DES* | *None* |
| *e.g. Server* | *CIS Benchmark (OS-specific)* | *Debug off, no default creds, no server banners, CORS allowlist* | *Document any exceptions* |
| *e.g. Container runtime* | *CIS Docker / Kubernetes Benchmark* | *Non-root containers, read-only root fs* | *Document any exceptions* |
| *e.g. Cloud account* | *CIS AWS / GCP / Azure Foundations* | *Per benchmark* | *Document any exceptions* |
| *e.g. Database* | *CIS MongoDB / PostgreSQL Benchmark* | *Auth enabled, no default ports, TLS required* | *Document any exceptions* |
| | | | |

### PR.PS-02: Software maintenance

Track patching timeframes and component lifecycle. Apply patches within the timeframe set in the vulnerability management plan. Replace end-of-life software before it becomes a liability.

| Component type | Patching approach | SLA (critical / high / medium) | EOL tracking |
|---|---|---|---|
| *e.g. OS packages* | *Automated via package manager* | *24h / 7d / 30d* | *Distro release calendar* |
| *e.g. Container base images* | *Rebuild image with updated base, deploy fresh* | *24h / 7d / 30d* | *Image tag lifecycle* |
| *e.g. Application dependencies* | *Dependabot / Snyk, pin versions, review before merge* | *7d / 14d / 30d* | *npm/NuGet deprecation notices* |
| *e.g. Managed services* | *Provider-managed* | *Provider SLA* | *Provider announcements* |
| | | | |

### PR.PS-03: Hardware maintenance

For cloud-hosted products, hardware lifecycle is provider-managed. Document any physical hardware you manage directly.

| Hardware | Owner | Maintenance approach | Replacement trigger |
|---|---|---|---|
| *e.g. Cloud compute* | *Provider-managed* | *Automatic (instance replacement)* | *Provider-initiated* |
| *e.g. Developer workstations* | *IT team* | *OS patching, disk encryption verified* | *EOL or failure* |
| | | | |

### PR.PS-04: Logging

Ensure every service and infrastructure component generates log records that are shipped to centralized infrastructure. Logs must be usable for incident investigation under [Detect](/docs/kilpi/functions/detect).

| Log source | Format | Destination | Retention | Sensitive data |
|---|---|---|---|---|
| *e.g. Application services* | *Structured JSON (timestamp, source, severity, context)* | *Centralized log platform* | *90 days* | *Redacted before writing* |
| *e.g. Infrastructure (cloud)* | *Provider format (CloudTrail, etc.)* | *Centralized log platform* | *1 year* | *N/A* |
| *e.g. Database* | *Database audit log* | *Centralized log platform* | *90 days* | *No query parameters logged* |
| | | | | |

### PR.PS-05: Software integrity

Prevent unauthorized or tampered software from running in your environments. Verify the source and integrity of everything that executes.

| Control | Scope | Implementation |
|---|---|---|
| *e.g. Approved base images* | *All containers* | *Internal registry, image signatures verified before deploy* |
| *e.g. Dependency pinning* | *All application code* | *Lock files committed, checksums verified in CI* |
| *e.g. Deployment gates* | *Production* | *Code review required, CI/CD pipeline only, no manual deploys* |
| *e.g. DNS filtering* | *All environments* | *Block known malicious domains* |
| | | |

### PR.PS-06: Secure development practices

Integrate security into the development lifecycle. These practices feed into the supply chain controls tracked under [GV.SC](/docs/kilpi/functions/govern/supply-chain-risk).

| Practice | When it runs | Tool / method | Failure action |
|---|---|---|---|
| *e.g. Code review* | *Every PR* | *Peer review, security-flagged changes get extra reviewer* | *Block merge* |
| *e.g. Static analysis* | *CI on every push* | *SAST tool (e.g. Semgrep, CodeQL)* | *Block merge on high/critical* |
| *e.g. Dependency scanning* | *CI on every push + scheduled* | *Dependabot / Snyk / Trivy* | *Block merge on critical, alert on high* |
| *e.g. Build integrity* | *CI/CD pipeline* | *Reproducible builds, signed artifacts* | *Block deploy* |
| *e.g. Environment separation* | *Always* | *Isolated dev/staging/prod, no prod creds in non-prod* | *Audit violation* |
