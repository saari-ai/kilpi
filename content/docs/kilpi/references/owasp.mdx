OWASP (Open Worldwide Application Security Project) produces freely available standards, guides, and tools for web application security. Two of its projects are directly relevant to our methodology: the OWASP Top 10 and the Application Security Verification Standard (ASVS).

The **OWASP Top 10** is a consensus-driven ranking of the most critical web application security risks. It is updated periodically based on vulnerability data from hundreds of organisations and serves as a baseline awareness document for development teams.

The **ASVS** goes deeper. It is a catalogue of testable application security requirements organised by verification level. Where the Top 10 identifies risk categories, ASVS provides the specific controls to address them.

## How it fits

OWASP primarily maps to CSF subcategories under **Protect** (<span style={{whiteSpace: 'nowrap'}}>PR.AA</span>, <span style={{whiteSpace: 'nowrap'}}>PR.DS</span>, <span style={{whiteSpace: 'nowrap'}}>PR.PS</span>, <span style={{whiteSpace: 'nowrap'}}>PR.IR</span>) and **Identify > Risk Assessment** (<span style={{whiteSpace: 'nowrap'}}>ID.RA</span>). It answers a specific question: what are the most common ways web applications fail, and what controls prevent those failures?

The chain works like this:

1. CSF subcategory defines the outcome (e.g., <span style={{whiteSpace: 'nowrap'}}>PR.AA-02</span> — identities and credentials are managed)
2. ISO 27001 Annex A provides the control (e.g., <span style={{whiteSpace: 'nowrap'}}>A.8.5</span> — Secure authentication)
3. OWASP provides the risk context and verification criteria (e.g., A07:2021 — Identification and Authentication Failures, ASVS V2 — Authentication)

## ASVS verification levels

ASVS defines three verification levels, each suited to a different risk profile. Higher levels require more rigorous controls.

| Level | Intended for | Approach | CSF alignment |
|---|---|---|---|
| L1 — Opportunistic | All applications | Automated testing and basic manual review. Addresses the most common and easily exploitable vulnerabilities. | Covers baseline <span style={{whiteSpace: 'nowrap'}}>PR.AA</span>, <span style={{whiteSpace: 'nowrap'}}>PR.DS</span>, <span style={{whiteSpace: 'nowrap'}}>PR.PS</span> subcategories |
| L2 — Standard | Applications handling sensitive data | Deeper manual testing, threat modelling, and secure design review. The target level for most business applications. | Extends into <span style={{whiteSpace: 'nowrap'}}>ID.RA</span>, <span style={{whiteSpace: 'nowrap'}}>DE.CM</span> subcategories |
| L3 — Advanced | High-value targets (financial, health, critical infrastructure) | Full source review, penetration testing, architecture analysis. Maximum assurance. | Full coverage of <span style={{whiteSpace: 'nowrap'}}>PR.*</span>, <span style={{whiteSpace: 'nowrap'}}>ID.RA</span>, <span style={{whiteSpace: 'nowrap'}}>DE.CM</span> subcategories |
